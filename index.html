RPC 与 Thrift 关系.

  1. RPC(Remote Procedure Call), 远程过程调用通信协议.

    Thrift Tutorial (Thrift 简洁教程, 例子完全thrift源码中tutorial.thrift)
    http://thrift-tutorial.readthedocs.io/en/latest/intro.html

    协议层: 提供了数据是序列化和发序列化.

    支持如下协议:
TBinaryProtocol(2进制)
TCompactProtocol(紧凑型)
TDenseProtocol(与TCompactProtocol类似, 剥离元信息, JAVA实现中不可用)
TJSONProtocol(Json)
TSimpleJSONProtocol(Json只写What?)
TDebugProtocol(文本格式)

    传输层:

    支持如下协议:
TSocket(阻塞IO传输)
TFramedTransport(使用帧来发送数据, 而外发送数据长度, 需使用非阻塞IO服务器)
TFileTransport(将传输写入文件, java未实现)
TMemoryTransport
TZlibTransport(Java 不可实现)

    Processor:

    从协议层读取数据, 通过加载进入的Handle来处理数据并输出.

    Server:

    TSimpleServer - A single-threaded server using std blocking io. Useful for testing.
TThreadPoolServer - A multi-threaded server using std blocking io.
TNonblockingServer - A multi-threaded server using non-blocking io (Java implementation uses NIO channels). TFramedTransport must be used with this server.

消息系统作用

解耦, 冗余, 扩展性, 灵活性&峰值处理能力(消息队列关键组件, 突发超负荷请求而崩溃), 可恢复性(处理消息进程挂掉….在系统恢复后被处理, 降低用户体验不舒适度), 发送保证, 顺序保证, 缓冲(这个看的蒙, 没关系继续开下去), 理解数据流, 异步通信.

概念
  1. Broker (这破玩意就应该是核心了, 上边一大堆都是它来保证的.)
  2. Topic (逻辑概念, 物理上, 不同的topic消息, 分开存储. 一个topic也可以存储与一个或多个broker上)总结: topic和物理上broker存储的位置, 屁关系没有.
  3. Partition (物理概念, 一个topic包含一个或多个partition, 创建topic时可以指定partition数量, 每个partition对应一个文件夹, 该文件夹下存储该partition的数据和索引文件.)
  4. Producer (作用和名字一样.)
  5. Consumer (consumer属于特定的consumer group, 可以为每个consumer指定group name, 如果不指定group name, 则consumer属于默认的group), 使用 consumer high level API时, 同一topic的一条消息只能被同一个consumer group 内的一个consumer消费, 但多个consumer group可同时消费这一消息.
  6. Zookeeper(和之前的RocketMq中的nameserver相似, 管理集群配置的)
Kafka 架构
  • 盗图一张 :)

  • 上游Producer, 管理集群配置Zookeeper, 消息队列Broker, 下游Consumer.
    管理集群配置Zookeeper(leader, rebalance)
    Producer push模式将消息发布到broker, Consumer使用Pull模式从broker订阅并消费消息.

  • Push Vs Pull
    简洁的解释就是.
    Push尽最大的可能满足发送方, 接收方收不收的了这就是问题.
    Pull尽最大的可能满足接收方, 如果接收方上游数据过多, 一定会出现数据积压问题.
    没有100%好的, 取决于业务场景.

  • Topic&Partition(郁闷, 看到这里凌乱了.)

    以上说的这些是把握给搞晕了, 怎么就一个逻辑就多个partition了, 怎么就提升效率了. 瞬间就凌乱了…

    4 判断终止条件.

递归问题的思想

  • 递归搞了好些…还是一本算法书上说的有道理
    你只不过再找这样的一个公式 F(n) = F(N-1) + F(N-2)
    还有一个终止条件.
    栈空间压入顺序: F(n)/F(n-1)/F(n-2)/F(n-3)……F(2)/F(1)
    栈出去的顺序: F(1)/F(2)…F(n-3)/F(n-2)/F(n-1)/F(n)

  • 以上程序对栈内存极具毁灭性. 我的8G内存, 10000左右报了栈溢出…
    心塞.

  • 递归调用产生了太多的冗余调用, 这个在python中通过装饰器可以节省计算.
    通过{fibonna(N):vaule}方式.

  • 如果你有兴趣, 减少冗余调用. 可以自己试一试, 反正我是过了…:)
